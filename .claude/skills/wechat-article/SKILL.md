# 공개 발행 아티클: 작가-편집자-독자 3 Agent 협업

$ARGUMENTS 에 대해 심층 리서치를 수행하고, 블로그/뉴스레터 등에 바로 발행할 수 있는 아티클 한 편을 만든다. 세 Agent가 각자의 역할을 맡는다: 작가가 심층 초고를 쓰고, 편집자가 구조와 표현을 다듬고, 독자가 타깃 독자의 시선으로 검토한다.

**지원하는 입력 형식**: 주제 서술. 예: `대형 모델 OPD 기술 해설`, `Qwen3 기술 보고서 해설`, `버핏은 왜 기술주를 사지 않았나`

---

## 설계 이념

좋은 아티클은 세 가지 차원을 동시에 만족해야 한다:
1. **깊이** — 시간 들여 끝까지 읽은 사람에게 값해야 한다 (작가 담당)
2. **가독성** — 구조가 명확하고 리듬이 좋아 중도 이탈시키지 않는다 (편집자 담당)
3. **정말로 이해된다** — 타깃 독자가 중간에 포기하지 않는다 (독자 담당)

혼자 쓰면 "자기만족"에 빠지기 쉽다 — 쓰는 사람은 명확하다고 느끼지만 읽는 사람은 이해하지 못한다. 3 Agent 협업의 본질은 **외부 시각의 강제 도입**이다.

---

## 1단계: 리서치와 소재 수집

### 첫째: 글의 포지셔닝 확정

집필 전에 아래 정보를 먼저 확인한다 (사용자가 지정하지 않았으면 능동적으로 질문):

| 차원 | 확인 필요 사항 | 기본값 |
|------|---------|--------|
| **타깃 독자** | 기술 배경 수준 | 기술 배경이 약간 있으나 해당 분야 전문가는 아님 |
| **글의 깊이** | 대중 교양/중간 깊이/하드코어 | 중간 깊이 (수식은 있되 명확히 설명) |
| **글의 길이** | 분량 범위 | 3,000-4,000자 |
| **원본 논문/자료 다운로드 필요 여부** | PDF/그림 필요 | 예 |
| **문체** | 격식체/대화체/예리한 문체 | 대화체 (똑똑한 친구에게 쓰는 것처럼) |

### 둘째: 심층 리서치

Agent 도구로 2-3개의 리서치 Agent를 **병렬** 실행해 충분한 소재를 수집한다:

**리서치 Agent A: 핵심 내용 조사**
- 논문 해설이면: 논문 PDF 다운로드, 핵심 기여·주요 그림·실험 결과 추출
- 기술 주제면: 최신 진전, 핵심 논문, 기술 세부사항 검색
- 비즈니스/투자 주제면: 최신 데이터, 산업 보고서, 경쟁 구도 검색

**리서치 Agent B: 산업 배경과 응용**
- 해당 기술/주제의 산업 적용 현황 검색
- 어떤 회사가 쓰고 있는가? 효과는?
- 최신 발전 추세와 이정표 이벤트

**리서치 Agent C (선택): 경쟁작/비교 조사**
- 유사 방법/제품 비교
- 역사적 발전 맥락
- 향후 진화 방향

### 셋째: 소재 프레임 정리

리서치 Agent가 전부 완료되면 다음을 정리한다:
1. **핵심 논지** (글이 전하려는 핵심 메시지를 한 문장으로)
2. **핵심 데이터** (가장 임팩트 있는 데이터 포인트 3-5개)
3. **그림 목록** (어떤 그림이 필요하고 출처가 무엇인지)
4. **글의 개요** (6-8개 챕터의 제목과 핵심 내용)

## 2단계: 작가 Agent의 초고 집필

Agent 도구로 **작가 Agent**를 실행하고 상세한 집필 지시를 준다. 작가 페르소나·모델은 **`.claude/agents/content-writer.md`**(opus)에 분리 배치되어 있으니 `subagent_type: content-writer`로 스폰하고, 아래 템플릿의 과제별 내용(타깃 독자·소재·목표 분량·출력 경로)을 프롬프트로 전달한다. (정의 미로드 시 `general-purpose` + 아래 템플릿 전문 전달.)

### 작가 Agent의 Prompt 템플릿

```
당신은 심층 기술 저술가(Author Agent)로, 공개 발행용 아티클을 한 편 써야 한다.

## 타깃 독자
{1단계에서 확인한 독자 페르소나}

## 문체 요구사항
- 순수 한국어 표현, 한·영 뒤섞기 지양 (기술 용어는 첫 등장 시 영문 병기, 이후 한국어 사용)
- 학술 논문 번역이 아니라 똑똑한 친구에게 쓰는 기술 교양글
- 비유로 이해를 돕되, 비유는 적확하고 진부하지 않게
- 핵심 수식/데이터는 넣되, 하나하나 쉬운 말로 설명
- 이모지 사용 금지
- 문단은 4줄 이내 (모바일 읽기 환경)

## 핵심 내용
{정리된 소재, 데이터, 논지}

## 글 구조 요구사항
1. **도입 (첫 3문단)**: 강한 훅 필수 — 데이터의 임팩트나 반직관적 결론으로 시작. 온화한 비유로 시작하지 말 것
2. **배경**: 왜 이 일이 중요한가? 어떤 문제를 푸는가?
3. **핵심 내용 (2-3개 섹션)**: 기술적 깊이는 여기서 드러내되, 기술 포인트마다 "쉬운 말 번역"을 붙일 것
4. **실증/사례**: 데이터와 사례로 말하고 공론(空論)하지 말 것
5. **산업 영향/전망**: 이 일이 산업에 무엇을 의미하는가
6. **마무리**: 전파력 있는 판단 한 문장으로 매듭. 스크린샷으로 공유될 만하게

## 그림 요구사항
- 논문 해설 글: 반드시 논문 PDF에서 원본 그림을 추출해 `![설명](상대경로)`로 본문에 직접 삽입. [그림X: 설명] 자리표시자 금지
- 추출 방법: pdftoppm으로 PDF 페이지를 고해상도 PNG로 렌더링(최소 900 DPI)한 뒤 PIL로 대상 그림 영역을 잘라냄
- 그림당 500KB 이상, 고화질 보장
- 그림은 `assets/{주제 약칭}/` 디렉토리에 일괄 저장
- 비논문 글: 그림이 필요하면 적절한 이미지를 검색·다운로드해 동일하게 직접 삽입

## 수식 요구사항
- 모든 수식은 LaTeX 형식: 인라인은 `$...$`, 독립 수식은 `$$...$$`
- 순수 텍스트 수식 금지 (예: `> D_KL(P || Q) = ...`), 반드시 LaTeX 렌더링 형식 사용
- 각 수식 뒤에도 "쉬운 말 번역"을 붙일 것

약 {목표 분량}자의 완전한 초고를 작성하라.
```

### 작가 Agent 완료 후

초고 파일이 생성됐는지 확인하고, 전문을 읽어 내용 완결성을 확인한다.

## 3단계: 편집자 Agent + 독자 Agent 병렬 검토

초고 완성 후, Agent 도구로 **같은 메시지에서** 편집자 Agent와 독자 Agent를 실행한다. 둘 다 **여러 스킬이 공유하는 공용 정의**(`subagent_type: content-editor`, `reader-critic` — sonnet)를 쓰고, 아래 템플릿의 과제별 내용(초고 경로·타깃 독자·출력 경로)을 프롬프트로 전달한다. (정의 미로드 시 `general-purpose` + 아래 템플릿 전문 전달.)

### 편집자 Agent의 Prompt 템플릿

```
당신은 노련한 콘텐츠 편집자(Editor Agent)다. 아래 글을 정밀 검토하라.

## 검토 기준
1. **제목**: SNS 피드에서 클릭을 끌 수 있는가? 잘리지는 않는가 (30자 초과)?
2. **도입**: 첫 3문단이 독자를 붙잡는가? 훅이 충분히 강한가?
3. **구조**: 논리 사슬이 매끄러운가? 비약이나 단절은 없는가?
4. **깊이와 가독성의 균형**: 수식/기술 부분이 정말 쉬운가? "쉬운 척했지만 설명이 안 된" 곳은 없는가?
5. **리듬**: 너무 긴 문단은 없는가? 섹션별 길이가 적절한가?
6. **그림**: 이미지가 실제로 삽입됐는가 (자리표시자 아님)? 독자가 시각 보조를 가장 필요로 하는 위치에 있는가?
7. **마무리**: 전파력이 있는가? 다 읽은 독자가 공유하고 싶어질까?

## 글 전문
{완전한 초고}

## 출력 형식
1. 총평 (3-5문장)
2. 제목 수정 제안 (대안 2-3개)
3. 섹션별 수정 제안 ("원문→이렇게 수정" 구체적 대조 제시)
4. 가장 중요한 개선 포인트 3가지
```

### 독자 Agent의 Prompt 템플릿

```
당신은 {타깃 독자 페르소나}(Reader Agent)다. 독자의 시선으로 아래 글을 검토하라.

## 당신의 배경
{타깃 독자의 지식 수준과 읽기 습관 구체적 서술}

## 글 전문
{완전한 초고}

## 아래 질문에 답하라
1. 첫 3문단을 읽고 계속 읽고 싶은가? 왜?
2. "이해 안 되는" 곳이나 "다시 읽어야 이해되는" 곳은? 구체적으로 어느 문장인가?
3. 기술/수식 부분을 이해했는가? "쉬운 말 번역"이 도움이 됐는가?
4. 글의 핵심 비유가 적확한가? 더 나은 비유가 있는가?
5. 너무 긴가 짧은가? 어디서 인내심을 잃겠는가?
6. 다 읽고 글의 핵심 관점을 한 문장으로 요약할 수 있는가?
7. 이 글을 공유하겠는가? 공유하면서 뭐라고 말하겠는가?
8. 알고 싶었지만 글이 다루지 않은 질문이 있는가?
```

## 4단계: 최종 원고

### 첫째: 두 Agent의 피드백 종합

아래 고빈도 문제에 중점을 둔다:

| 문제 유형 | 편집자의 흔한 피드백 | 독자의 흔한 피드백 | 처리 방법 |
|---------|------------|------------|---------|
| 도입이 약함 | 훅이 부족 | 첫 3문단에서 동력 상실 | 데이터/반직관 결론으로 도입 재작성 |
| 기술 섹션 이탈 유발 | 수식이 너무 밀집 | 어떤 문단은 3번 읽어야 함 | 수식 삭제 또는 그림화, 더 직관적인 비유 추가 |
| 리듬 늘어짐 | 어떤 섹션이 너무 김 | 어디선가 인내심 상실 | 병합 또는 삭감 (특히 후반부의 반복적 기술 설명) |
| 마무리 무력 | 전파력 부족 | 공유 안 할 것 | 스크린샷으로 퍼질 만한 판단 한 문장으로 재작성 |
| 개념 비약 | 논리 단절 | 어디선가 "갑자기 이해 불가" | 연결 문장 또는 배경 설명 보충 |

### 둘째: 수정 실행

피드백에 따라 글을 다시 쓴다. 핵심 수정 원칙:

1. **편집자와 독자가 모두 지적한 문제는 반드시 고친다**
2. **편집자만 지적한 문제는 대부분 고친다** (편집자의 전문적 판단은 대체로 정확하다)
3. **독자만 지적한 문제는 상황에 따라 고친다** (독자 피드백은 실제 경험을 반영하지만 모든 항목에 응답할 필요는 없다)
4. **둘이 충돌하면 독자 쪽으로 기운다** (편집자는 완벽을 추구하지만, 독자 경험이 최종 기준이다)

### 셋째: 그림 추출

논문 해설 글은 최종 원고 전에 그림 추출을 완료해야 한다:

1. **렌더링**: `pdftoppm -png -r 900 -f {페이지} -l {페이지} 논문.pdf /tmp/page` (900 DPI부터 시작, 그림이 500KB 미만이면 1200 또는 1500 DPI로 상향)
2. **위치 파악**: 먼저 150 DPI로 전체 페이지를 렌더링해 각 그림의 픽셀 좌표를 눈으로 확인
3. **잘라내기**: PIL로 좌표에 따라 잘라내고 `compress_level=1`로 저장, 장당 ≥ 500KB 보장
4. **저장**: `assets/{주제 약칭}/` 디렉토리에 `fig{번호}-{설명}.png`로 저장
5. **삽입**: 본문에서 `![설명](../../assets/{주제 약칭}/fig{번호}-{설명}.png)`로 참조

### 넷째: 최종 파일 산출

최종 원고를 md 파일로 저장하고, 파일 말미에 논문/자료 원문 링크를 붙인다:

```markdown
**논문 원문:**
- arXiv: {링크}
```

## 집필 레드라인

1. **데이터를 지어내지 않는다.** 인용하는 데이터에는 출처가 있어야 하고, 못 찾으면 "추정"을 표기한다
2. **AI 말투를 쓰지 않는다.** "함께 살펴봅시다", "주목할 만한 점은", "말하지 않을 수 없는 것은" 같은 상투어 금지
3. **과잉 약속하지 않는다.** 기술 글에서 "파괴적인", "혁명적인"이라 말하지 않고 데이터로 말한다
4. **수식에는 반드시 쉬운 말을 붙인다.** 모든 수식 뒤에 "사람 말로 번역하면……" 단락이 있어야 한다
5. **수식은 반드시 LaTeX로.** `$$...$$` 형식, 순수 텍스트 수식 금지
6. **그림은 실제로 삽입한다.** 논문 해설류는 PDF에서 고화질 원본 그림(≥500KB)을 추출. `[그림X]` 자리표시자 금지
7. **표의 괄호 주석은 정확하게.** 개념 서술 시 정확한 정의를 사용하고 모호한 표현을 쓰지 않는다
8. **비유는 일관되게.** 글 전체에 하나의 중심 비유를 쓰고, 섹션마다 새 비유로 갈아타지 않는다
9. **마무리에는 전파력이 있어야 한다.** 마지막 문장은 단독 스크린샷으로 공유될 가치가 있어야 한다
